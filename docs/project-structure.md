# بنية المشروع المقترحة

هذا الهيكل مصمم لتطبيق ويب عربي لإدارة السوبر ماركت مع قابلية التوسع لاحقاً إلى عدة فروع ونقاط بيع.

## المجلدات الحالية

```text
frontend/
  src/
    lib/              # منطق مشترك مثل حسابات الكاشير وقراءة باركود الميزان
    pages/            # الصفحات الرئيسية مثل الصفحة العامة وواجهة POS
    assets/           # الصور والموارد الثابتة
  public/             # ملفات عامة للمتصفح

backend/
  src/
    config/           # إعدادات البيئة والتكوين
    data/             # طبقة التخزين والمستودعات مع fallback بين PostgreSQL والذاكرة
    routes/           # تعريف مسارات API
    modules/
      auth/           # المصادقة والصلاحيات
      users/          # المستخدمون والأدوار
      products/       # الأصناف والفئات والباركود
      inventory/      # المخزون والحركات والدفعات والصلاحية
      sales/          # المبيعات والفواتير والمرتجعات
      purchases/      # المشتريات والموردون
      accounting/     # القيود والحسابات والتقارير المالية
      crm/            # العملاء والديون والإشعارات
      hr/             # الموظفون والحضور والانصراف
      reports/        # التقارير ولوحات القيادة
    shared/
      db/             # طبقة الوصول إلى قاعدة البيانات
      middleware/     # middlewares مشتركة مثل المصادقة والتدقيق
      utils/          # أدوات مساعدة مشتركة
    app.ts            # تكوين تطبيق Express
    server.ts         # نقطة تشغيل الخادم

docs/
  database-schema.sql # مخطط قاعدة البيانات الأولي
  project-structure.md
```

## ملاحظات معمارية

- الواجهة الأمامية تعتمد React + Vite + Tailwind مع RTL كلغة أساسية.
- شاشة POS يجب أن تظل مستقلة عن بقية الواجهة لتسهيل العمل دون اتصال لاحقاً.
- backend الحالي يستخدم repository layer واحدة، ويمكنه العمل على PostgreSQL عند توفر DATABASE_URL أو الرجوع تلقائياً إلى الذاكرة في بيئات العرض السريع.
- كل حركة مخزنية يجب أن تنتج من مستند واضح: بيع، شراء، مرتجع، تسوية، أو هالك.
- الدفعات وتواريخ الصلاحية يجب أن تكون جزءاً من طبقة المخزون نفسها وليس خاصية إضافية على المنتج فقط.
- تعدد الفروع يبدأ من وجود branch و warehouse في كل مستند وحركة مخزون.
- RBAC يجب أن يُطبق على مستوى الـ API وليس الواجهة فقط.
- دعم العمل دون اتصال في POS لاحقاً سيكون عبر Local Storage أو IndexedDB مع Sync Queue.
